所以它本质上是：

类型	你的项目是否属于
通用问答系统	不是

所以你不用焦虑别人都写问答系统。问答系统只是最常见、最容易包装的项目形态，但不是唯一形态。

你的项目如果包装得好，反而更像真实业务系统。

三、第三优先级：DPO 做主线，GRPO 做亮点实验

你的文档里已经写了后续高优先级是验证 DPO 真实收益，等训练栈升级后再评估 ORPO / OnlineDPO / GRPO；GRPO 目前被放在更激进的低优先级路线里。

我的建议是：

主线继续做 DPO

DPO 很适合你的项目，因为你的错误类型很容易构造 chosen / rejected：

chosen：字段正确、JSON 合法、技能无误报
rejected：方向错、职责漏抽、技能幻觉、字段混乱

你可以优化 build_preference_dataset.py，让它不只是从预测结果生成偏好对，而是支持多种 rejected 来源：

模型错误输出
规则扰动输出
历史 badcase 输出
人工构造错误输出

比如故意制造 rejected：

把 AI应用开发 改成 算法工程；
往技能里加 CUDA / TensorRT / PyTorch；
把加分项误放到必备技能；
把职责和任职要求混在一起；
删除最后一条核心职责。

这样 DPO 的训练信号会更明确。

GRPO 作为实验模块

你有最多 9 张 L20，可以做 GRPO，但不要让它喧宾夺主。

建议新增：

src/jobmatch_tune/rl/rewards.py
src/jobmatch_tune/train/train_grpo.py
scripts/train_qwen3_7b_grpo_smoke.sh

reward 直接围绕你的字段指标：

reward =
JSON 合法性
+ schema 完整性
+ 岗位方向准确率
+ 核心职责 F1
+ 必备技能 F1
+ 经验/学历匹配
- 技能幻觉惩罚
- 输出冗余惩罚

GRPO 最好先在 Qwen 7B / 8B 上做 hard case 实验，14B 只做小规模验证。

简历里可以写成：

在 DPO 偏好优化基础上，设计字段级 reward，探索 GRPO 对岗位方向边界和技能误报 hard case 的强化效果。

不要写“GRPO 显著提升效果”，除非你有大评估集结果。

四、第四优先级：补真实简历解析能力

你现在主链路更偏 JD。后续如果要做成人岗匹配系统，必须补简历侧。

建议构建一个简历 schema：

{
  "目标岗位": "",
  "教育背景": [],
  "核心技能": [],
  "项目经历": [

      "项目名称": "",

    }

然后准备三类数据：

注意：简历数据一定要去隐私化，不要保存真实姓名、手机号、邮箱、身份证、学校具体个人信息。

八、不要优先做这些

下面这些不是没价值，而是不适合你现在马上做：

1. 不要优先做全参数微调

你的项目资源约束下原本就选择了 LoRA / QLoRA，不直接上 70B+，这个路线是合理的。

2. 不要优先做重型 DeepSpeed / FSDP

文档里也把 DeepSpeed ZeRO / FSDP 放在低优先级。 除非你要做多机工程训练展示，否则收益不如补评估和产品功能。

3. 不要把项目硬改成问答系统

你的优势是结构化抽取和匹配，不是聊天问答。最多加一个“解释层问答”，比如：

为什么这个 JD 被判定为 AI 应用开发？
为什么我的简历匹配度只有 72？

不要把主线改成“招聘问答机器人”。

推荐你最终按 4 个阶段推进
第 1 阶段：产品闭环，1 周

目标：补人岗匹配。

要做：

/api/match
match_prompt
match_score.py
前端人岗匹配 Tab
样例 JD + 简历

产出：

用户能输入 JD + 简历，看到匹配分、优势、缺口、优化建议。
第 2 阶段：评估闭环，1-2 周

目标：扩大评估集，验证 DPO。

要做：

50 条 eval → 200/500 条 eval
真实简历 eval
JD-resume match eval
SFT vs DPO 对比报告
技能幻觉率统计

产出：

一份完整 benchmark 表。
第 3 阶段：后训练增强，1-2 周

目标：DPO 做扎实，GRPO 做亮点。

要做：

优化 preference dataset
构造 hard rejected
字段级 reward
7B/8B GRPO smoke
SFT/DPO/GRPO hard case 对比

产出：

DPO 真实收益报告 + GRPO 探索报告。

目标：让项目像真实系统。

批量解析

产出：

一个可以演示、可以部署、可以写进简历的完整 LLM 应用工程。
最终推荐路线

你的后续优化主线应该是：

JD 解析
  ↓
简历解析
  ↓
人岗匹配
  ↓
DPO 偏好优化
  ↓
字段级评估
  ↓

  ↓

一句话总结：

先把项目从“结构化抽取 demo”升级成人岗匹配产品；再扩大评估集验证 DPO；最后用字段级 reward 做 GRPO 亮点实验。不要一开始追 GRPO 或更大模型，真正能提升项目含金量的是：用户场景完整、评估可信、服务可用、后训练路线讲得清楚。
