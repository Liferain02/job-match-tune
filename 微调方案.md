# JobMatchTune 技术方案

## 1. 项目目标

实现一个面向招聘场景的小模型微调实践项目，支持对公开 JD 和合成/脱敏简历进行结构化解析，并输出岗位匹配分析。

核心目标不是做完整招聘系统，而是做一个可复现、可评估、适合低 GPU 资源的小模型微调闭环：

1. 导入或采集公开 JD。
2. 清洗 JD 和简历文本。
3. 构造指令微调数据集。
4. 先做 Base Model 基线评估。
5. 使用 Qwen3 系列小模型做 QLoRA / LoRA 微调。
6. 对比微调前后在 JD 解析、简历解析、匹配分析上的效果。
7. 使用 Gradio 做轻量演示页面。

项目边界：

1. 不做账号体系。
2. 不做完整企业招聘推荐平台。
3. 不使用真实未脱敏个人简历。
4. 不绕过招聘网站登录、验证码或反爬。
5. 不追求训练大模型，只验证“小模型 + 领域数据 + 可量化评估”的工程闭环。

------

## 2. 可行性结论

当前方案可行，但需要按低资源路线收敛：

1. 模型选择需要更新：`Qwen/Qwen2.5-1.5B-Instruct` 已不适合作为 2026 年新项目默认首选。它仍可作为低风险兼容备选，但主线建议切到 Qwen3。
2. 默认推荐 `Qwen/Qwen3-1.7B` 或 `Qwen/Qwen3-4B-Instruct-2507`。前者更省显存，后者指令跟随和结构化输出更强。
3. 微调方式建议默认使用 QLoRA，而不是普通 LoRA。普通 LoRA 更简单，但显存占用更高；GPU 少时应优先保证能跑通。
4. 数据量不宜一开始设到 1000-2000 条/任务。首轮建议使用 300-800 条高质量样本验证有效性，再扩到 2000-5000 条。
5. 训练目标要聚焦。首轮不要同时追求三个任务都显著提升，优先做“JD 解析”和“简历解析”，匹配分析可以先用规则评分 + 模型解释生成。
6. 评估必须先于训练完成。没有 Base Model 基线和固定测试集，微调效果不可证明。

推荐实践路线：

```
阶段 1：不训练，只跑 Base Model + 规则后处理，建立评估集和指标。
阶段 2：只训练 JD 解析 + 简历解析，使用 Qwen3 + QLoRA，小样本快速验证。
阶段 3：加入匹配分析样本，或保留规则评分，只让模型生成解释。
阶段 4：扩充数据、调参、做 Gradio 演示。
```


可选组件：

```
Playwright：仅当静态 HTML 无法获取公开 JD 时使用。
Unsloth：可选，用于进一步降低 QLoRA 训练显存和提升速度，但首版不强依赖。
vLLM / SGLang：只用于推理服务或演示，不用于首轮训练。
DeepSpeed：低资源单卡阶段不建议引入，复杂度高于收益。
```

模型建议：

```
默认低资源：Qwen/Qwen3-1.7B
效果优先：Qwen/Qwen3-4B-Instruct-2507
兼容备选：Qwen/Qwen2.5-1.5B-Instruct
```

低资源原则：

1. 默认使用 QLoRA 4-bit。
2. 默认 `max_seq_length=1024`，只有长 JD/长简历任务再升到 2048。
3. 默认 LoRA rank 使用 `r=8`，先验证收益，再考虑 `r=16`。
4. 不合并 LoRA 权重也可以演示，优先保存 adapter。
5. Qwen3 普通模型默认有 thinking mode；结构化 JSON 抽取任务应关闭 thinking 或使用 Instruct-2507 这类 non-thinking 模型，避免输出 `<think>...</think>` 干扰 JSON 解析。

### 3.1 模型选型建议

| 模型 | 推荐场景 | 优点 | 风险 |
|---|---|---|---|
| `Qwen/Qwen3-1.7B` | 8-16GB 显存、快速验证 | 小、快、上下文 32K、Qwen3 主线 | 默认可能输出 thinking 内容，需要显式关闭 |
| `Qwen/Qwen3-4B-Instruct-2507` | 16-24GB+ 显存、效果优先 | 指令跟随更强，non-thinking，适合 JSON 输出 | 训练显存和时间高于 1.7B |
| `Qwen/Qwen2.5-1.5B-Instruct` | 老环境、依赖版本受限 | 成熟稳定，生态兼容好 | 不是当前主线，效果上限较低 |

推荐决策：

1. 如果 GPU 很少，先用 `Qwen/Qwen3-1.7B` 跑通全流程。
2. 如果有 16GB 以上显存，优先试 `Qwen/Qwen3-4B-Instruct-2507` 的 QLoRA。
3. 如果服务器环境无法升级到 `transformers>=4.51.0`，再退回 Qwen2.5。
4. 简历项目展示时，建议写“Qwen3 + QLoRA”，比 Qwen2.5 更符合当前模型生态。

### 3.2 依赖版本建议

`requirements.txt` 建议固定下限，不要完全不锁版本：

```text
torch>=2.3
transformers>=4.51.0
accelerate>=0.34
datasets>=2.20
peft>=0.12
trl>=0.10
bitsandbytes>=0.43
pydantic>=2.7
pandas>=2.2
scikit-learn>=1.5
beautifulsoup4>=4.12
gradio>=4.44
pyyaml>=6.0
pytest>=8.0
ruff>=0.6
```

说明：

1. `transformers>=4.51.0` 是 Qwen3 的关键下限，版本过低会无法识别 `qwen3` 模型类型。
2. `trl` + `peft` 是当前 SFT + LoRA/QLoRA 的主流组合，可以减少自写训练循环的复杂度。
3. `bitsandbytes` 用于 4-bit QLoRA；如果 CUDA/驱动不兼容，要优先解决环境，而不是退回全量训练。
4. `pydantic` 用于定义和校验 JD/简历/匹配结果 schema，比散落的 dict 更可维护。

## 4. GPU 资源建议

### 4.1 推荐配置矩阵

| 资源情况 | 推荐模型 | 推荐方案 | max_seq_length | batch size | gradient accumulation | 说明 |
|---|---|---|---:|---:|---:|---|
| 8GB 显存 | Qwen3-1.7B | QLoRA | 512-1024 | 1 | 16-32 | 能做小样本验证，速度较慢 |
| 12-16GB 显存 | Qwen3-1.7B / Qwen3-4B-Instruct-2507 | QLoRA | 1024 | 1 | 8-16 | 推荐起步配置 |
| 24GB 显存 | Qwen3-4B-Instruct-2507 | QLoRA / LoRA | 1024-2048 | 1-2 | 8 | 较稳妥 |
| L20 / 48GB | Qwen3-4B-Instruct-2507 / Qwen3-8B | LoRA / QLoRA | 2048 | 2-4 | 4-8 | 可以做完整实验 |

### 4.2 显存不足时的降级顺序

按以下顺序处理，不要一开始换复杂方案：

1. 开启 4-bit QLoRA。
2. 将 `max_seq_length` 从 2048 降到 1024，再降到 768 或 512。
3. 将 `per_device_train_batch_size` 设为 1。
4. 增大 `gradient_accumulation_steps` 保持有效 batch。
5. 开启 `gradient_checkpointing`。
6. 关闭 Qwen3 thinking mode，减少无用输出 token。
7. 减少 LoRA target modules，只训练注意力层。
8. 减少训练样本，先跑 100-300 条 smoke test。

```

RAG
Agent
LangChain

```

爬虫要求：

1. 只抓公开页面。
2. 不绕过登录、验证码、反爬。
3. 设置请求间隔。
4. 保存来源、抓取时间、岗位链接、岗位名称、公司名称、地点、薪资、岗位描述、任职要求。
5. 原始 HTML 可选保存；如果保存，注意不要保存 Cookie、Token 或账号信息。

### 6.2 公开数据集补充

可引入公开候选人-岗位匹配数据集作为冷启动数据，但不要直接依赖英文/合成数据证明中文招聘场景效果。

用途：

1. 匹配评分规则设计参考。
2. 简历字段 schema 参考。
3. 少量翻译/改写后生成中文样例。

限制：

1. 数据分布可能与中文招聘 JD 差异较大。
2. 合成数据容易让模型学到模板，而不是稳定抽取能力。
3. 公开数据集必须检查许可证。

```

```

要求：

1. 不使用真实个人隐私数据。
2. 电话、邮箱、学校、公司、人名都使用虚构或脱敏内容。
3. 样例中保留技能、项目、实习、教育等任务相关信息。

## 8. 任务设计

首轮微调只建议保留 2 类抽取任务，匹配分析先做规则评分。

### 8.1 任务 A：JD 结构化解析

输入：

```text
请解析以下招聘 JD，抽取岗位方向、核心职责、必备技能、加分项、经验要求和学历要求。
只输出 JSON，不要输出解释。

JD：
{jd_text}
```

输出：

```json
{
  "岗位方向": "AI应用开发",
  "核心职责": [
    "负责大模型应用功能开发",
    "建设 RAG 或 Agent 应用能力",

  "必备技能": ["Python", "大模型 API 调用", "RAG", "后端开发"],

}
```


低资源首版不建议把匹配分数完全交给模型学习。推荐：

1. 使用规则函数计算匹配分数和匹配等级。


数据划分：

```

```

首轮建议数据量：

```
JD 解析：300-800 条
简历解析：200-500 条

```

扩展阶段数据量：

```
JD 解析：1000-3000 条
```

质量要求：

1. `test.jsonl` 必须人工抽查，不能和训练集重复。
2. 同一 JD 改写样本不能跨 train/test。
3. 标签中不确定的字段用空数组或空字符串，不要编造。
4. JSON key 保持固定，不要同义字段混用。

## 10. 数据清洗与标签构造

### 10.1 JD 文本清洗

实现文件：`src/jobmatch_tune/preprocess/clean_text.py`

规则：

1. 去除 HTML 标签、脚本、样式、导航栏、页脚。
2. 合并连续空白字符。
3. 删除手机号、邮箱、微信号等隐私字段。
4. 删除“立即沟通”“投递简历”等网页按钮文本。
5. 删除重复段落。
6. 保留岗位职责、任职要求、加分项、福利等核心段落。

### 10.2 去重

实现文件：`src/jobmatch_tune/preprocess/deduplicate.py`

策略：

1. 对 `clean_text` 做 SimHash / MinHash。
2. 文本相似度超过 0.9 视为重复。
3. 同公司、同岗位、同城市、文本高度相似时只保留一条。
4. 训练集和测试集之间也要去重。

### 11.4 训练命令

先做 100 条 smoke test：

```bash
python -m jobmatch_tune.train.train_lora --config configs/train_qlora.yaml --max_train_samples 100 --max_eval_samples 50
```

确认能跑通后再完整训练：

```bash
python -m jobmatch_tune.train.train_lora --config configs/train_qlora.yaml
```

训练实现要求：

1. 使用 `tokenizer.apply_chat_template` 拼接 messages。
2. 只对 assistant 部分计算 loss。
3. 支持 QLoRA 和 LoRA 两种模式。
4. 支持 `max_train_samples` 和 `max_eval_samples` 便于低成本调试。
5. 保存 adapter 权重和训练日志。
6. 保存 valid loss 最优 checkpoint。
7. 固定随机种子。
8. 对 Qwen3 普通模型暴露 `enable_thinking` 配置，结构化抽取默认关闭。
9. 推荐支持 TRL `SFTTrainer`，但保留自定义 collator，确保只训练 assistant 输出。
10. QLoRA 可优先使用 `target_modules: all-linear`；如果显存不足，再降级为 `q_proj/v_proj`。

## 12. 推理与后处理

实现文件：

```
src/jobmatch_tune/inference/predict.py
src/jobmatch_tune/inference/postprocess_json.py
```

要求：

1. 输入 `task_type`。
2. 根据 `task_type` 选择 prompt 模板。
3. 支持 Base Model 和 LoRA/QLoRA adapter。
4. 输出原始文本。
5. 尝试解析 JSON。
6. 如果 JSON 解析失败，做简单修复。
7. 修复后仍失败则标记 `parse_error`。
8. Qwen3 thinking 模型需要过滤 `<think>...</think>` 内容，或在 chat template 中设置 `enable_thinking=False`。

JSON 修复策略：

1. 提取第一个 `{` 到最后一个 `}` 之间的内容。
2. 替换中文引号为英文引号。
3. 删除尾随逗号。
4. 尝试 `json.loads`。
5. 失败则返回 `raw_output`。

## 16. 主要风险与优化建议

### 16.1 低 GPU 风险

风险：显存不足、训练慢、调参成本高。

优化：

1. 默认 QLoRA。
2. 首轮只跑 100-300 条 smoke test。
3. 先用 `max_seq_length=1024`。
4. 先训练 1-2 epoch，看验证集是否改善。
5. 保留 Base Model + 规则后处理作为对照。

### 16.2 数据质量风险

风险：规则标签噪声大，模型学到错误格式。

优化：

1. 固定 JSON schema。
2. 每类任务人工抽查至少 50 条。
3. 测试集全部人工抽查。
4. 对低置信标签不训练，或标为空值。

### 16.3 匹配分析主观性风险

风险：匹配分数本身不唯一，模型容易编造。

优化：

1. 分数由规则函数产生。
2. 模型只生成解释和建议。
3. 评估中加入“是否基于输入”的人工检查。

### 16.4 微调收益不明显

风险：Base Model 本身已能完成任务，小样本微调提升有限。

优化：

1. 重点看 JSON 正确率、字段稳定性、技能 F1，而不是主观感觉。
2. 增加负样本和边界样本，例如非 AI 岗位、技能缺失、长 JD、短简历。
3. 对 prompt、schema、后处理和数据质量做 ablation。

## 17. 最小可行版本

如果 GPU 很少，建议 MVP 只做这些：

1. 100 条 JD 解析训练样本。
2. 100 条简历解析训练样本。
3. 50 条固定测试集。
4. QLoRA，`max_seq_length=768`，训练 1 epoch。
5. 只比较 JSON 正确率、技能 F1、字段完整率。
6. 匹配分析用规则评分 + Base/QLoRA 生成建议。

这个版本就能证明：

1. 数据构造链路跑通。
2. 小模型微调能提升结构化输出稳定性。
3. 低 GPU 资源下方案可复现。

1. Qwen 官方在 2025-04-29 发布 Qwen3，包含 `Qwen3-0.6B`、`Qwen3-1.7B`、`Qwen3-4B`、`Qwen3-8B`、`Qwen3-14B`、`Qwen3-32B` 等 dense 模型，以及 MoE 模型；官方说明 Qwen3 在训练数据、推理、指令跟随、Agent 和多语言能力上相对 Qwen2.5 有明显升级。
